# 用户档案更新

## 触发条件

触发条件（满足任一即触发）：

1. **重要对话结束**：counseling 对话产生了新的诊断信息（维度变化、综合体变化、关键事件），由 counseling skill 或 Agent 判断调用
2. **用户直接要求**：用户明确要求更新档案
3. **历史材料整合**：用户提供年末总结、月度素描、回溯分析等历史材料，需写入档案系统

日常闲聊不触发。如果材料中各部分内容不清楚具体时间，跟用户交流确认。

## 工作目录

用户档案位于 workspace 根目录下：`user_profile/`。以下所有档案路径均以此为根。
避免错误：只搜了 ma-zhuang/user_profile/，就宣布该文件夹不存在

## 流程

> **前置**: 凡涉及心理分析（周期文件分析区、overview 分析区、交互因果链合成），必须确保 counseling 框架已在当前上下文中。首次分析前通过 `Skill("counseling")` 加载；同一会话后续分析复用已加载框架。仅在以下情况重新确认：分析深度跃迁（周→月/季/年）、触及新的维度或交互因果链判断、距上次框架确认超过 10 块。若框架已不在上下文中，重新加载。详见底部"更新原则"。

### 第一步：判断是否值得更新

按 `_cycle-file-rules.md` 二c 的入库筛选原则判断。不满足则不更新，直接结束。

### 第二步：加载现有档案

- 读 `user_profile/last-update.md`，获取上次更新日期
- 读 `user_profile/comprehensive/overview.md` 了解当前综合状态
- 按需读 dim1/dim2/dim3 的 overview 文件
- 读 `_interaction-chain-examples.md` 了解跨时间结构性规律，用于指导交互因果链的判断

目的：在现有快照基础上增量更新。如果相关档案已在当前上下文中，跳过文件读取。

### 第三步：回溯检测缺失的周期文件，逐级补写

不依赖天数计算，而是检查已过去的周期是否已有对应文件。补写原则：**逐级向上，下级为源**。

数据源层级：`对话记录 → 周记 → 月记 → 季记 → 年记`。上级从下级摘录事实和合成分析。下级缺失则先级联补下级。

### 历史材料：按 `_cycle-file-rules.md` 多时间层规则处理

**周记补写**：
- 数据源：对话记录（当前上下文）+ 当日用户自评
- 按 `_cycle-file-rules.md` 周记格式：事实区（按日）+ 分析区（单日分析 + 本周综合分析）
- 分析前：加载 counseling skill
- 无数据标注"（无记录）"，不编造

**月记/季记/年记补写**（统一规则，下级缺失则级联补下级）：

| 周期 | 前置 | 数据源 | 跨周期参考 |
|------|------|--------|-----------|
| 月记 | 该月所有周记 | 该月全部周记（comprehensive + dim1-3） | 上月月记 + 上季季记 |
| 季记 | 该季所有月记 | 该季全部月记 | 上季季记 + 上年年记 |
| 年记 | 该年所有季记 | 该年全部季记 | 上年年记 |

事实区：月记摘录转折点/突破/新认知框架/维度级变化；季记/年记摘录主要矛盾级变化。均保留日期和原编号，不是所有下级事实都向上传导。
分析前：加载 counseling skill。
分析区：合成跨下级交互因果链 + 聚合综合体动态形成本级变化；对上周期延续来的链标注延续节点。跨周期加载按 `_cycle-file-rules.md` 规则3。

级联示例：7月1日发现6月月记缺失 → 先检查6月周记(W23-W26) → W26缺失则先从对话记录补写W26 → 周记齐全后读4份周记 → 摘录事实 → 合成分析 → 写6月月记。

无记录时段统一标注"（无记录）"，不编造。

### 第四步：更新周期文件（如触发）

#### 4.0 跨周期参考加载

按 `_cycle-file-rules.md` 规则3的加载对照表，加载跨周期参考文件（上个同级 + 上个上级），加载 counseling skill 进行跨周期框架分析。

四个位置，按序写入：
1. `four-dimensions/dim1-elements/{period}/{file}.md`
2. `four-dimensions/dim2-links/{period}/{file}.md`
3. `four-dimensions/dim3-karma/{period}/{file}.md`
4. `comprehensive/{period}/{file}.md`

所有周期文件遵循**事实/分析分离**规则：上半区用户事实（只追加，不删除，编号），下半区 Agent 分析（可引用事实编号），中间以”---”分隔。

**分析区强制章节**（月记/季记/年记，缺一不可）：

| 章节 | 内容 | 月记 | 季记 | 年记 |
|------|------|------|------|------|
| **四维度分析** | 维1五要素/维2链接/维3业障/维4心灵 | 月内有变化才写 | 季内有变化才写 | 必写 |
| **五综合体变化** | 感受/认知/涵容/能量/力比多 | **必写** | **必写** | **必写** |
| **主要矛盾分析** | 当前周期的主要矛盾及变化趋势 | **必写** | **必写** | **必写** |
| **交互因果链** | 跨维度因果链，标注方向、理由、证据来源 | ≥1条 | ≥1条 | ≥2条 |

无显著变化写”—“而非不写。完整格式模板见 `_cycle-file-rules.md`。

**月记/季记/年记必须包含 frontmatter**，声明 `sources`（覆盖的下级周期及其最后更新日），格式见 `_cycle-file-rules.md`。创建或更新时同步维护 `date` 和 `sources`。

**年记特有规则**：
- **分析区不分 H1/H2 或 Q1-Q4 子章节**。所有分析（五综合体、交互因果链、主要矛盾）以全年为粒度。半年度/季度分段破坏年度综合视角。
- **禁止在已有分析区末尾追加新小节而不重读上文**——这会导致"H1分析+H2补丁"的拼贴而非真正的年度综合（第五步已提及，此处再次强调针对年记）。

- 跨期评论，比如2026.1.1评论了2023.5.7的日记，则该评论本身如果按照第一步要求判断值得更新，应该作为事实，增加到评论日（即2026.1.1）的日记中

### 4.5 更新范围验收（防截断错误）——仅批量操作（≥5个文件）

批量操作（≥5个文件）时，多文件更新前显式列出完整文件清单（dim1/dim2/dim3/comprehensive 各周期文件 + overview，写全路径），更新后逐项核对。

- **comprehensive 做完了 ≠ 任务做完了。** dimension 文件不会自动更新。
- **月度做完了 ≠ 季度做完了。** 每个周期层级独立检查。
- **凭感觉判断"差不多做完了"不可信。** 回到清单逐项核对。

### 第五步：向上传导（强制步骤，不可跳过）

写完任一周记/月记/季记后，立即检查上级文件。传导表：

| 刚写完 | 检查上级 | 触发条件 | 操作 |
|--------|---------|---------|------|
| 周记 | 当月月记 | 本周有非寻常变化 | 月记不存在→新建；月记已存在→读月记分析区，结合新事实重新综合 |
| 月记 | 当季季记 | 本月的非寻常变化涉及五要素、四维度或五综合体的变化 | 同上 |
| 季记 | 当年年记 | 本季存在非寻常变化，且该变化涉及五要素、四维度或五综合体的结构性改变时触发 | 同上 |

**"非寻常变化"** = 不是每天都会发生的事。日常吃饭、看书、遛弯不算。无则只追加事实到上级，不重写分析。

**月→季、季→年增加筛选**：非寻常变化是否触及五要素（学业/事业/社交/身体/心理心灵）、四维度（链接/业障/心灵）或五综合体（感受/认知/涵容/能量/力比多）的变化？一周的情绪波动不往上推，拿到offer（事业维）、涵容扩容（五综合体）、业障命名（业障维）等触及结构骨架的才传导。

**重新综合、有机整合，而不是机械追加。** 

向上传导必须在完整的上下文中进行，不能分段、分次、分批。

1. 如果上级不存在，Agent 会基于下级来综合创建它。
2. 如果上级已经存在，Agent 必须阅读上级的全部、现有的内容，然后将新的下级材料的综合纳入现有的框架中。禁止在已有分析区末尾追加新小节而不重读上文——这会导致"H1分析+H2补丁"的拼贴而非真正的年度综合。
3. 如果新的数据没有改变上级报告的整体判断，则只追加事实，不重写分析。

传导不是机械复制——上层只摘录下级中构成上层结构性变化的部分。一次传导一级，不越级。

**批量操作场景（如 diary-process）**：如果一次会话中写了 ≥3 个周记文件，允许在所有周记写完后统一执行向上传导。但必须：
- 对每个涉及的月份，检查月记是否需新建/更新
- 对每个涉及的季度，检查季记是否需新建/更新
- 对每个涉及的年度，检查年记是否需新建/更新
- **显式列出受影响的上层周期清单，逐个打勾确认**
- 读取每个上层文件的 `sources` frontmatter——缺失的下级周期或过期日期一目了然，不需要凭记忆判断
- 禁止"凭感觉觉得差不多都传导了"

### 第六步：更新 overview 文件 ⚠️ 批量操作最易遗漏——完成后回到此处核查

覆写以下 overview 文件中有变化的时间层级：
1. `comprehensive/overview.md`
2. `four-dimensions/dim1-elements/overview.md`
3. `four-dimensions/dim2-links/overview.md`
4. `four-dimensions/dim3-karma/overview.md`

更新方式：
- 覆写分析区前，加载 counseling skill
- 周更新：追加周度变化条目，更新最近关键变化列表
- 月更新：追加月度变化条目，下层有结构性意义的向上传导
- 季更新：追加季度变化条目
- 上层周期（年/季）只在有结构性变化时修订，不机械每周重写

**覆写前对比机制**（逐层，从细到粗）：

| 层级 | 旧值 | 新值 | 覆写条件 |
|---|---|---|---|
| 周度 | 现有 overview 周度层 | 刚写完的周记综合分析 | 本周主要矛盾/变化与旧值不同 |
| 月度 | 现有 overview 月度层 | 刚写完的月记（或各周记聚合） | 月度趋势方向改变 |
| 季度 | 现有 overview 季度层 | 刚写完的季记（或各月记聚合） | 季度级主题/方向改变 |
| 年度 | 现有 overview 年度层 | 各季度聚合 | 年度主要矛盾转移 |
| 近三年 | 现有 overview 近三年层 | 各年度聚合 | 三年窗口内核心叙事改变 |

####模板参考
- 模板见 `_overview-templates.md`。Overview 文件同样遵循**事实/分析分离**规则。
- 交互因果链撰写时参考 `_interaction-chain-examples.md`；撰写完成后，如果有重要的交互模式发现，补充进这个文件，如有需要可以优化修改此文件原有表述。

#### 注意

- 下层变化不一定触发上层覆写——只有改变上层主要矛盾或趋势方向时才传导
- 旧 overview 某层缺失或格式不一致，以新值为准

## 核心规则：事实与分析分离

所有档案文件（周期文件 + overview 文件）统一遵循：

- 文件分上下两区，用 `---` 分隔
- **上半区 · 用户事实**：Agent 只追加，不删除。修改需经用户同意。每条事实标注具体日期，用字母做序号标记（如 `a.`、`b.`、子项 `a.1` `a.2`）；用户自己的关键分析，也是事实的一部分，以**用户分析**标记。
- **下半区 · Agent 分析**：可引用事实编号简述，如"引用事实 c 用户晕倒后冷静分析"，避免长篇复述
- **层级递进**：周记事实最全面；月记/季记/年记只摘录最重要事实，保留原始字母编号
- **时间锚定**：每条事实必须标注具体日期；周记按日分节，分析区单日分析也按日对齐

详细格式规范见 `_cycle-file-rules.md` 和 `_overview-templates.md`。

> **工具提示**: 创建新周期文件时，Write 工具可能要求先 Read。若遇此限制，用 Bash `touch path` 创建空文件后 Read 再 Write。Bash heredoc 仅限创建新文件，更新已有文件必须 Edit。

## 更新原则

适用于所有文件：
- **不覆盖历史**：overview 覆写是快照更新，周期文件是追加式记录，两者不冲突
- **引用原话**：关键变化描述尽量引用用户原话作为证据
- **简洁**：每条快照 1-2 句，不写论文
- **定性不定量**：不评分数，不打数字分。用"强/中/弱""通畅/阻滞""直接性/对抗性"等定性描述
- **不猜测**：分析基于已有记录中的事实进行综合。缺失日期的状态不编造（标注"无记录"）。交互因果链的箭头必须同时满足时间先后和概念因果，不从零散记录强行构造。
- **与用户自述一致**：用户说"心力不错"就不能写"心力低迷"，可在自述基础上补充分析视角但不能推翻
- **禁止裸分析**：凡涉及对用户内容进行心理分析（周期文件分析区、overview 分析区、交互因果链合成、跨周期对比），必须加载 counseling skill。不允许仅凭模板词汇做表层描述。上下文已有档案时 counseling 无需重读已加载文件。同一会话中连续处理多个日记块时，首次加载 counseling 后可在上下文中复用。仅在以下时机重新确认框架仍在上下文中：
  - 从周记分析升级到月记/季记/年记综合（分析深度跃迁）
  - 触及新的维度或交互因果链判断
  - 距上次确认已超过 10 块且即将进入分析步骤
  如果框架已不在上下文中，重新加载。

## Optimizer触发
- 如果系统内找不到 ma-zhuang-optimizer skill，不进行此项目。
- 本次对话中，如果本skill自上次 ma-zhuang-optimizer 分析本skill之后（首次则从对话开始算），累计被调用 ≥5 次，则在完成当前所有步骤后，调用 ma-zhuang-optimizer 分析本skill。
- 或者由 diary-process 路由进入且处理块数≥5，完成全部步骤后触发 ma-zhuang-optimizer 分析本skill。
