# 日记导入处理

## 触发条件

用户在对话框或指定文件夹提供日记材料，要求导入 profile 系统。

**输入类型**：单一大文件（docx/txt/md）/ 多小文件（Notion/Obsidian 导出）/ 对话贴文。

## 工作目录

原始文件保留在 `diary/`，处理中间产物在 `diary/_process/{材料名}/`。

## 流程

分两步：**扫描结构 → 生成处理计划** → **按计划逐项执行**。

### 第一步：预处理

1. 文件转换为 md（保留标题层级）。**依次尝试**：
   - `markitdown`（Python 包，支持 docx/pptx/pdf/xlsx/html）
   - python-docx（仅 docx 纯文本提取）
   - pandoc（通用转换器）
   
   当前工具不可用则 `pip install` 或 `winget install` 安装，安装失败则尝试下一个。

2. 图片 → `![desc](path)`，不处理图片语义
3. 转换产物输出到 `diary/_process/{材料名}/`

4. **按周拆分（强制）**：转换后的 md 若超过 **2000 行**，必须以周为单位拆分为独立文件。拆分规则：
   - **优先**：扫描 `### M.D-M.D` 格式的周边界，每遇到一个周标题，将该周内容抽取为独立文件 `{材料名}_W{周号}.md`
   - **降级**：无 `### M.D-M.D` 标题时（如 Obsidian/Notion/纯文本导出），扫描行首日期模式（`M.D` / `YYYY.M.D` / `M月D日`）
   - **再降级**：仍无法识别日期边界时，按自然段落边界 + 行数阈值（~200行/块）粗拆分，在计划中标注"无标题拆分，边界为估算"
   - 非日记内容的文章：根据文章标题中的日期归入对应周的文件，追加在日记块之后
   - 月度素描、季度/年度总结：另存为独立文件（如 `_01月素描.md`）
   - 拆分后，每份小文件约 200-800 行，agent 一次只面对一周的工作量
   
   **拆分原理**：大文件（>2000行）会在心理上压倒 agent，诱发"量大赶工"→抄近道→壳子文件。按周拆分后每次只处理一周，agent 不会因总量恐慌而滑坡。粗拆分的边界可能跨越自然周，但后续 profile-update 会按实际日期分配到正确周期文件。禁止跳过此步骤。

### 第二步：结构扫描，生成处理计划

扫描拆分后的小文件列表，按时间排序。第一遍只读每个文件的标题（`###` 头）和首尾几行，确认日期范围和内容类型，不读全文。生成 `_processing-plan.md`：

```
# 处理计划 — 2023H1

来源：日记2023H1.docx → 拆分为周文件

## 待处理
- [ ] [日记块] 2023H1_W01.md — 1.1-1.8 元旦第一周 (~600字)
- [ ] [日记块] 2023H1_W02.md — 1.9-1.15 (~800字)
- [ ] [日记块·续] 2023H1_W03a.md — 1.16-1.19 (~1800字)
- [ ] [日记块·续] 2023H1_W03b.md — 1.20-1.22 (~1500字)
- [ ] [文章] 2023H1_W03_文章.md — 1.16 社交思考 (~300字)
...
- [ ] [月度素描] _01月素描.md
```

**计划生成规则**：每个拆分后的文件对应一个计划项。文件<200字的与相邻文件合并。文件路径中标注了所属周（如 `W01`）——断点恢复和进度跟踪依赖文件路径，不是行号。

**二次拆分（切分上限）**：即使已按周拆分，单份文件若仍超过 **2000 字**，必须继续按以下规则二次拆分：
- ≤2000 字，≤连续一周（通常已是单周，按日或主题边界下刀）
- 单日日记超过 2000 字，仍然整体作为一个子块交给 profile-update（单日是自然单元）
- 单日有多篇独立文章时，每篇文章独立为 `[文章]` 块
- 拆分后的文件名加后缀：`W03a.md`、`W03b.md`

**任务量预估**：计划生成后，统计总共文件数和估计处理时间，主动告知用户。量大不构成跳过逐块调用或降低分析质量的理由——质量优先于速度，文件已拆分消除了"量大恐慌"的借口。

**文章处理规则**：文章按写作日期归入对应周记的事实区，作为 `补充·文章` 子节。文章长度超过2000字的，在周记中摘录核心论点（≤200字），不全文搬运。文章标题列表在月度总结中统一收录。

### 第三步：按计划逐项执行

**打勾标准（强制执行）**：每项打勾前必须确认四条件：
1. 对应的周期文件存在且非空
2. 事实区有具体内容（不是"待从原文补充"），按日组织的具体事件，引用用户原话
3. 分析区有基于 counseling 框架的分析（不是"待补充"），五综合体必须逐一描述（感受/认知/涵容/能量/力比多），不可只写一个词或一句话
4. **本块的 profile-update skill 调用已返回确认**（通过 Skill 工具调用，非 diary-process agent 自行写入）

四项全过才能打勾。**壳子文件禁止打勾。打勾即宣誓以上 4 条均已满足。** 如果当前环境无法完成完整分析（如上下文不足），标记为失败而非打勾糊弄。

对"待处理"中每一项（从上到下）：

1. 读取当前块对应的拆分文件（不加载其他文件）
2. 调用 profile-update skill，喂原文
3. profile-update 自行完成完整流程：加载 counseling → 判断入库 → 写周期文件 + 分析 → 更新 overview → 更新 last-update
4. 等 profile-update 全部完成并返回确认。若调用超时或无响应，视为单条交割失败，按"错误处理"节处理（标记失败→继续下一项→全部完成后重试）。
5. 该项移到"已完成"，打勾
6. 下一项

**硬规则——禁止抄近道**：
- **逐块完成**：禁止把多个块攒在一起"批量写事实、事后统一分析"。每个块必须当场完成完整的 profile-update 流程
- **禁止脚本化分析**：禁止用任何形式的脚本或批处理替代逐块的手动心理分析。包括但不限于：bash 循环生成分析文本、Python 调用 LLM 批量写分析区、模板填充式生成五综合体描述。分析必须经过 counseling skill 加载后的框架视角——脚本不可能加载 counseling 框架。**脚本可用于机械操作**：文件格式转换、编码转换、按标题边界拆分、字符统计——这些不涉及心理分析的操作，脚本是最优工具。写入 `user_profile/` 的周期文件永远不能通过脚本创建。
- **禁止批量创建文件**：严禁用任何批处理方式（bash for 循环、heredoc、cat <<EOF、echo > 重定向、Python 文件操作等）一次性创建多个周期文件。一经发现，该批次所有文件视为无效壳子，需全部重做。**禁令原理：逐块调用 profile-update 是 diary-process 唯一的合规写入路径；任何批量创建 = 绕过了 counseling 加载 = 壳子文件**
- **不越权**：禁止 diary-process agent 自行直接写 profile 文件。必须通过 `Skill("profile-update")` 工具调用，由 profile-update skill 自行完成文件写入。diary-process 只做切分和送达，不越权做 profile-update 的工作。**每写入一个周期文件前自检：这个 Write/Edit/Bash 调用是我（diary-process agent）发出的，还是 profile-update skill 发出的？前者=违规。Bash 工具写 user_profile/ 同样违规——不因工具不同而改变性质。**
- **不干涉**：profile-update 内部自行处理周期合成（周→月→季→年）和向上传导，diary-process 不得干涉或跳过。diary-process 也不得代替 profile-update 做逐月检查点——那是 profile-update 第五步的职责

- **禁止自行决定暂停执行。** 所有块处理完毕或用户明确要求暂停前，不间断执行。

- **分析质量远大于处理效率。** 禁止 agent 在处理过程中冒出"加速""求效率""批量处理以节省时间"等质量滑坡信号。一旦出现此类念头，视为 agent 正在被工作量压垮、试图抄近道——应立即暂停，向用户报告当前状态，而非继续赶工。慢不是问题，壳子才是。

**框架复用规则（批量模式专属）**：diary-process 首次处理前调用 `Skill("profile-update")` 加载完整框架。此后若 counseling 和 profile-update 框架均在上下文中，后续块直接在当前框架下执行完整流程（事实→分析→五综合体→交互因果链），不重复调用 Skill 工具。仅当：分析深度跃迁（周→月/季/年）OR 距上次框架确认超 10 块 OR 用户中断后恢复，才重新调用 Skill 工具确认框架。**框架复用 ≠ 降低分析质量**——分析仍须含五综合体逐一描述 + 引用原文。

**每 5 块自审计（强制）**：每处理完 5 个块，暂停并逐条核查以下问题：
1. 过去 5 块是否每块都完成了完整分析（五综合体逐一描述）？
2. 过去 5 块的周期文件分析区是否有实质性内容（不是一句话）？
3. 过去 5 块是否出现过"量大赶工"的念头？如果有，在哪一块、被什么压住了？

任何一条答"否"或"有"——立即停止，报告用户自审计结果。**发现问题后的处理协议**：若发现质量滑坡，回退重做最近 5 块（自审计失败的块不得打勾）。若连续两次自审计发现同类问题，剩余所有块暂停，交由用户决策——不要硬撑出更多壳子。

### 第四步：验收

**职责边界**：逐月验收是 diary-process 的职责（检查文件存在+内容充实+sources 一致）。向上传导（创建/更新月记分析区+季记+年记）是 profile-update 第五步的职责。两者不重叠——diary-process 验收结构完整性，profile-update 负责分析综合。

**逐月检查点**：每处理完一个自然月的所有日记块+文章后，立即验收该月：
1. 该月所有周记文件存在、事实区有内容、分析区有内容
2. 该月月记已创建/更新
3. 月记 `sources` 中列出的周记与实际已创建的周记一致

三项全过才进入下个月。禁止所有月堆积到最后统一验收。

**最终验收**：
1. `_processing-plan.md` 中全部项已打勾（且满足第三步的打勾标准）
2. `user_profile/` 中目标时间范围的周期文件已生成（检查 `comprehensive/weekly/` 下文件覆盖目标区间）
3. 处理计划中所有 `[文章]` 和 `[月度素描]` 项已处理（整合入对应周记/月记）
4. `diary/_process/{材料名}/` 目录已清理（全量处理完成后删除）

## 中间文件管理

```
diary/
├── 日记2023H1.docx              # 原始（永久保留）
├── _process/                     # gitignore
│   └── 2023H1/                   # 完成后删除整个目录
│       ├── 日记2023H1.md         # 转换产物（处理结束后删除）
│       ├── 2023H1_W01.md         # 拆分产物——第一周（处理结束后删除）
│       ├── 2023H1_W02.md         # 拆分产物——第二周（处理结束后删除）
│       ├── ...                   # 所有拆分文件（处理结束后删除）
│       ├── _01月素描.md          # 拆分产物——月度素描（处理结束后删除）
│       └── _processing-plan.md   # 处理计划（处理结束后删除）
```

**全量处理完成后**，删除 `diary/_process/{材料名}/` 整个目录。仅保留原始 docx 文件在 `diary/` 下。

> Windows 清理: `rm -rf` 在 Windows 下偶发 "Device or resource busy"。降级方案：`rm -rf` 失败 → `rm -f` 逐文件 → `rmdir` 空目录。

## 错误处理与断点恢复

### 转换失败
当前工具转换失败 → 依次尝试第一步的降级方案。全部失败 → 报告用户文件名和错误类型。用户选择：重试 / 手动提供纯文本 / 跳过。

### 单条交割失败
单次 profile-update 调用未正常完成 → 该项保留 `[ ]` 标记，追加注释 `<!-- 失败: YYYY-MM-DD, 原因简述 -->`。继续处理下一项。全部完成后汇总失败项，重试。二次重试仍失败则交由用户处理。

### 用户中断
用户要求中断 → 当前项不标记完成。`_processing-plan.md` 已保存进度，下次继续。

### 断点恢复
重新打开 session → 读 `diary/_process/{材料名}/_processing-plan.md`。文件不存在 → 新导入。文件存在 → 从"待处理"第一个 `[ ]` 继续，已打勾项不重复处理。
